Executive Summary

Lineage: DragonClaw is a security-hardened fork of MetaClaw (v0.3). We identified and tested cascading hallucination vulnerabilities and defense gaps in the original MetaClaw, then built DragonClaw by upgrading it with a 3-tier defense stack, defense-gated conversation memory, auto-spawn session chaining, and 160 adversarial tests across 22 suites. References to "MetaClaw" in competitor comparisons refer to the original upstream project.

DragonClaw combines three architectural elements: tiered conversation memory retrieval, defense-gated fact verification on retrieved results, and automatic session chaining with context-window monitoring. This report maps the competitive landscape across open-source projects, academic papers, enterprise platforms, and patent filings.

Key Finding: No widely adopted system integrates all three. Individual components exist in isolation. The combination — particularly defense-gated retrieval treating stored memory as untrusted — represents a genuine architectural gap.
Closest Competitor: TierMem (2026) — provenance-aware tiered memory with verified write-back. Gets close to (a)+(b) but lacks automatic session chaining (c).
Patent Landscape: Active but fragmented. No clearly identified patent claims the exact combination of defense-gated memory retrieval + auto session chaining. IP whitespace appears to exist.

Recommended Actions

  • 1
    Publish the defense-gated retrieval architecture. This is the strongest novel claim. No mainstream framework implements it. MeVe and A-MemGuard validate the concept academically — DragonClaw validates it with production test data.
  • 2
    Benchmark against MemGPT/Mem0/Zep on recall, cost, and poison resistance. No canonical benchmark exists for adversarial memory recall across sessions — define the standard.
  • 3
    File provisional patent on the combined three-part architecture if IP protection is desired. The IP landscape is fragmented with no clear incumbent claim on the combination.
  • 4
    Build FactVerifier V2 (embedding-based) to close the O79 keyword-matching gap. Current V1 is vulnerable to paraphrase attacks where poison text contains truth aliases.
  • 5
    Create a public adversarial memory benchmark. No canonical benchmark exists for RAG memory poisoning across sessions. 160 tests with poison propagation data is a strong foundation.
Bottom Line: The three ingredients (tiered retrieval, defense gating, auto session chain) exist individually across the ecosystem. Nobody has integrated all three — especially not on local models at zero API cost with adversarial test data proving the architecture works. That gap is real, documented, and strategically meaningful.
